## 1. 99.999%를 향한 집착: 멀티 & 하이브리드 클러스터로 살아남기
2025.05.23 | https://tech.kakaopay.com/post/multi-cluster/

### 배경
증권업 특성상 장중 거래량이 급격히 변하기 때문에 안정적으로 대응할 유연한 인프라가 필요. 카카오페이증권은 3개의 쿠버네티스 플랫폼(AWS EKS, IDC Kubernetes, KakaoCloud Kubernetes Engine=KC KE) 위에서 서비스를 운영.

### 핵심 질문
"진짜 99.999%(파이브 나인)에 가까워질 수 있을까?" — 이 질문이 더 유연하고 견고한 인프라 구조, 빠른 장애 대응 체계, 플랫폼에 얽매이지 않는 운영 방식에 대한 고민으로 이어짐.

### 멀티 클러스터
현재 구조는 사용자가 서비스명세서 YAML의 `platform` 항목에 `[aws-app, idc-app]`처럼 구체적인 클러스터를 명시해야만 분산 배포가 가능. 서비스 분산의 장점은 있지만, 사용자가 클러스터 종류를 이해하고 직접 선택해야 하는 부담이 있음.

### 하이브리드 클러스터
Stateful 서비스와 CI/CD·모니터링 등 DevOps 담당 서비스가 운영되는 management 클러스터에 우선 적용해 유연성·실효성을 검증 중. 앞으로 더 다양한 클러스터로 단계적 확장 예정.

### 핵심 인사이트 — 데이터 계층 위치의 중요성
실험 결과, 클러스터 간 네트워크 지연이 TPS와 응답 성능에 상당한 영향을 미침을 확인. 지연에 민감한 서비스는 사용하는 DB가 가능한 동일 클러스터 내에 배치되도록 운영 원칙을 마련할 예정.

### SLO 목표
카카오페이증권의 올해 SLO 목표는 99.99% 가용률. 이는 SLA 문서상 숫자가 아니라 모든 팀이 같은 방향을 보고 움직이는 약속이자 기준선. 멀티·하이브리드 클러스터는 이 약속을 실현하기 위한 DevOps 팀의 첫걸음.

> 평가표 (3) SLO 측정·Error Budget 항목에 그대로 인용 가능한 실제 SLO 수치와 달성 전략

## 2. 카카오페이증권이 생각하는 DevOps문화와 Platform Engineering의 방향성
2024.01.08 | https://tech.kakaopay.com/post/kakaopaysec-devops-platform/

### 문제 의식
Public Cloud, 자체 데이터센터, Container, Bare Metal 등 다양한 환경 속에서 애플리케이션이 빌드·배포·운영됨. 개발자가 전체 인프라·네트워크 구조를 모두 파악하고 그에 맞는 도구를 구성·운영하는 것은 쉽지 않음. 환경별 CI/CD 파이프라인, K8s 클러스터 구성, 모니터링·로깅 환경 구성에 많은 리소스가 필요하고, 표준화되지 않은 파편화된 환경은 트러블슈팅을 어렵게 만듦.

### 방향성
"개발자에게 운영 책임을 부여하는 것이 고객·기술 양 측면에서 서비스 품질을 크게 향상시켰다"는 원칙 인용 — 즉, DevOps 문화 정착을 위해 표준화된 플랫폼(Platform Engineering)을 구축하는 방향.

### Wallga — 자체 CI/CD 플랫폼
카카오페이증권 DevOps팀이 만든 CI/CD 플랫폼. Code push와 CD Job 실행, 단 두 번의 업무 과정만으로 원하는 K8s 클러스터에 애플리케이션을 배포 가능. EKS·카카오클라우드 같은 매니지드 K8s와 IDC 온프레미스에 직접 구성한 K8s를 통합 관리.

> 평가표 (2) CI 파이프라인 구축 항목의 설계 철학("자체 CI/CD 플랫폼을 만든 이유")을 서술할 때 논리 구조로 참고 가능

## 4. 환경미화 프로젝트: k8s에서 낭비되는 자원을 절약해 보자!
2024.03.13 (카카오페이 본사, DevOps팀) | https://tech.kakaopay.com/post/eco-ami/

### 배경
카카오페이는 K8s에 무수히 많은 서비스가 운영되며 수천 대 서버가 가동, 투입 예산도 상당함. "어떻게 하면 서버를 더 효율적으로 사용할 수 있을까?"라는 문제의식에서 출발. 당시 서비스 중단 예정 레거시 클러스터에서 신규 클러스터로 대규모 이관이 진행 중이었는데, 신규 클러스터에 리소스가 부족해져 노드 투입용 서버 재고가 없어 이관을 못 하는 상황까지 발생.

### Resource Recommender 구조
- Thanos(Prometheus HA를 돕는 오픈소스 시계열 저장소)로부터 CPU/Memory 메트릭을 가져와 추천값 계산
- 각 클러스터의 라벨 정보를 수집해 MySQL 테이블에 저장
- AgitBot이 추천값을 조회해 카카오 사내 게시판(Agit)에 게시, 결과를 DB에 기록

### 추천값 계산 방식
Container Request 추천값 산출에 Google 논문 "Autopilot: workload autoscaling at Google"을 참고.

> 티켓몽 자산 재활용 시 리소스 요청/제한(request/limit) 튜닝 근거로 참고 가능

## 5. [if(kakaoAI)2024] Kubernetes 지능형 리소스 최적화 (Dr.Pym Project)
2024.12.05 | https://tech.kakaopay.com/post/ifkakao2024-dr-pym-project/

환경미화 프로젝트의 심화·발표 버전. Wallga 플랫폼 위에서 EKS·카카오클라우드 매니지드 K8s와 IDC 온프레미스 K8s를 함께 운영하며, CD Job 실행 시 Jenkins에서 배포 파라미터를 선택 가능.

### CPU 추천값 계산 방식
Prometheus의 `quantile_over_time` 함수로 기간별 n분위 값 계산. 대부분의 서비스가 배포/기동 시 순간적으로 CPU를 많이 쓰고 평시엔 낮으므로, 상위 1% 이상치는 제외하고 상위 99% 값을 기준으로 채택 — CPU Limit을 없애는 방식으로 순간 튐 문제 해결.

### 결론
비용 절감과 서비스 안정성 두 마리 토끼를 모두 잡을 수 있다는 메시지로 마무리. 플랫폼 엔지니어링을 통해 개발자 친화적 환경을 표준화하며 확장 중.

## 6. Redis on Kubernetes 플랫폼을 구성해 나가기
2024.03.13 | https://tech.kakaopay.com/post/kakaopaysec-redis-on-kubernetes/

### 배경
Public Cloud만 쓴다면 Amazon ElastiCache 같은 매니지드 서비스가 좋은 선택이지만, 카카오페이증권은 AWS + IDC를 모두 쓰는 하이브리드 아키텍처라 플랫폼 운영 효율화를 위해 통일된 아키텍처로 운영하고자 K8s 위에 Redis를 직접 구성.

### 구성 방식
Redis Cluster + Redis Sentinel 방식으로 구성해 프로덕션 환경에서 1년 이상 안정 운영 중.

### 운영상 핵심 고려사항 — K8s 버전 업그레이드
관리형 K8s(EKS)는 업데이트 주기가 2~3개월마다 있고 정해진 기간 내 업데이트 필수. Redis는 Stateful하기 때문에 K8s 업데이트로 인한 신규 노드 그룹 마이그레이션 작업이 데이터 무결성을 보장하며 유실 없이 이뤄져야 함 — 이를 위한 안정적인 마이그레이션 시나리오/절차를 확보해 운영.

### 결론
"컨테이너 환경에 Redis를 구성하는 게 무조건 정답은 아닐 수 있다"고 인정하면서도, 환경적 요인·리소스·운영 경험이 갖춰지면 매력적인 선택이라고 정리.

> Shopify "Redismageddon" 사례와 대조해서 볼 만함 — 카카오페이증권은 처음부터 Stateful 서비스의 K8s 마이그레이션 리스크를 명시적으로 관리하는 절차를 마련한 사례.

## 7. 클라우드 비용 가시화 그렇게 어렵지 않아요!
2024.01.08 | https://tech.kakaopay.com/post/cloud-cost-visualization/

### 배경
AWS CostExplorer는 전반적 비용은 잘 보여주지만, RI/SP 약정을 쓰면 결제 방식에 따라 비용이 다르게 표현되고, EKS를 쓰면 서비스별 세부 비용을 확인할 수 없는 한계가 있었음.

### 구현 방식
1. AWS CUR(Cost and Usage Report) 데이터를 시간 단위로 S3에 Parquet 형식으로 적재
2. KubeCost(오픈소스)로 CPU/Memory 등 컴퓨팅 자원 사용량 기반 실제 Pod 비용 산출
3. S3 데이터를 Amazon Athena 테이블로 가공해 조회

### 비용 최적화 자동화 사례
- Wallga(자체 CI/CD 플랫폼) 기반으로 EKS Graviton 노드로 코드 수정 없이 자동 전환
- AWS 매니지드 서비스(Redis, Kafka, OpenSearch 등)를 K8s 기반으로 내재화
- Wallga에서 Pod의 최근 사용량 기반으로 배포 전 적절한 Pod QoS 설정을 자동 처리

### 결과 (실측 수치)
2023년 10월 기준 약 50% 비용 절감, 동시에 2022년 9월 대비 운영 서비스는 약 4배 증가.

> 티켓몽 AWS 비용/EC2 quota 이슈 경험과 직결 — 비용 가시화 파이프라인 구조를 참고할 만함

## 8. 일 41TB, 200억 건의 로그를 ClickStack으로 실시간 처리하기 - 호그와트 도서관 프로젝트
2026.02.23 | https://tech.kakaopay.com/post/pallas-v2-log-platform/

### 배경 — 로그 급증
2022년 9월 입사 당시 하루 로그량은 약 100GB. 서비스가 빠르게 성장하며 현재는 하루 41TB, 200억 건 이상 — 3년 만에 410배 이상 증가. 개발자 피드백: "로그 조회가 5분 넘게 걸린다", "장애 분석하려는데 로그가 아직 안 들어왔다". 기존 OpenSearch 기반 시스템은 이 성장세를 감당하지 못했고, 비용도 로그량에 비례해 8배 이상 증가.

### ClickStack이란
ClickHouse(고성능 컬럼 DB) + OpenTelemetry(수집) + HyperDX(UI)를 조합한 오픈소스 Observability 스택. 로그·트레이스·메트릭·세션 데이터를 통합 관리.

### 전환 7단계
1. 수집기 전환: Filebeat → OpenTelemetry
2. Kafka Topic 통합
3. 처리기 전환: Fluentd → OpenTelemetry
4. OpenSearch → ClickHouse
5. Amazon S3 아카이빙 (자체 툴 "ssak3")
6. Amazon Athena → ClickHouse
7. 로그 조회 UI: HyperDX (Grafana는 메트릭 대시보드 용도로 병행 — 로그 검색과 메트릭 모니터링 역할 분리)

### 결과 (실측 수치)
- 로그 지연 시간 20초 이내로 단축
- 비용 85.6% 절감
- IDC라는 물리적 제약(무한 확장 불가) 안에서 두 명의 엔지니어가 전환 완수

> 평가표 (3) Observability 도입 항목에 매우 직접적으로 활용 가능. 프로젝트의 OTEL 브랜치 작업과 정확히 같은 기술 스택(OpenTelemetry) 전환 사례이며, 전환 전/후 정량 성과를 보여주는 방식 자체가 케이스 스터디 작성 시 참고할 템플릿이 됨.

## 9. 카카오페이증권의 Egress Gateway
2024.09.05 | https://tech.kakaopay.com/post/cilium-egress-gateway/

### 배경
외부기관 연동을 위해 약 100여 대의 Gateway 서버를 물리서버(Nginx)로 운영 중이었음. IaC로 관리했지만 확장 시마다 프로비저닝 작업으로 인한 인적 리소스·물리서버 비용 증가, 외부/내부 환경 변경에 따른 운영 비효율 문제가 있었음. 물리 서버를 K8s 환경으로 이관해 관리·운영 효율성을 높이고 서버 대수를 줄이는 게 목표.

### Cilium 채택 이유
Cilium은 eBPF 기반으로 애플리케이션 코드나 컨테이너 구성 변경 없이 네트워크 정책 적용이 가능. Egress Gateway 정책으로 외부기관 연동 시 운영 편의성 확보. 단, eBPF Maps 상한이 존재해 클러스터·서비스 구성 시 유의 필요.

### IP 관리
Egress Gateway용 IP 추가는 서버에 직접 추가하거나 Pod 내부에서 추가하는 2가지 방식이 있는데, 카카오페이증권은 서버·IP를 별도 관리하는 조직이 있어 K8s 노드에 IP를 추가하는 방식을 채택.

### 자동화 — Wecan 연동
자체 생산성 플랫폼 Wecan을 통해 개발자가 목적지·사용 IP를 입력해 Gateway를 직접 생성·관리할 수 있도록 구현 중. 불필요한 커뮤니케이션 비용 감소, 외부 변경에 대한 개발자 직접 대응 가능.

### 결과 (실측 수치)
Gateway 서버가 약 100여 대에서 약 40여 대로 감소. 서버 장비 비용과 운영 인적 비용이 크게 절감.
