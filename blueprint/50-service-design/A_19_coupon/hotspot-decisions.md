# Context 쿠폰 Hotspot 결정 기록

## 역할

`HOTSPOT.A.19-01~09`의 결정 상태와 공통 기준을 관리한다. BC와 요구사항은 업무 의미를, 도메인·영속성·서비스·API 문서는 각 책임에서 필요한 적용 규칙만 설명하고 이 문서의 결론을 다시 정의하지 않는다.

## 결정 상태

| Hotspot | 상태 | 결정 |
| --- | --- | --- |
| `HOTSPOT.A.19-01` | 확정 | 요청 직후에는 `발급 대기`, `UserCoupon` 생성과 수량 확정 뒤에는 `발급 완료`로 표시한다. 접수·대기·완료·실패를 구분한다. |
| `HOTSPOT.A.19-02` | 확정 | 주문 적용 시 쿠폰을 사용 예약으로 잠그고, 결제 최종 확정 사건을 검증한 뒤에만 사용 완료로 전환한다. |
| `HOTSPOT.A.19-03` | 확정 | 확정 실패·취소는 즉시 해제하고 결과 불명확 상태에만 짧은 유예를 적용한다. 회수 뒤 재사용은 검증된 취소·환불, 남은 유효기간, 캠페인 상태와 운영 중지를 모두 확인한 경우에만 허용한다. |
| `HOTSPOT.A.19-04` | 확정 | 발급된 쿠폰은 발급 당시 정책 버전, 진행 중 예약은 검증 당시 정책 버전을 고정한다. 새 정책은 새 발급·예약부터 적용하고 긴급 중지는 운영 제어로 처리한다. |
| `HOTSPOT.A.19-05` | 확정 | 판매자 전액 부담·자기 범위·승인된 템플릿은 판매자 권한으로 요청할 수 있다. 플랫폼·공동 부담, 제휴, 템플릿 초과, 고액·대량 보상은 운영 승인을 요구한다. CS 보상은 `caseRef`, 사유와 주문 또는 인시던트 참조가 필수다. |
| `HOTSPOT.A.19-06` | 확정 | 대량 대상은 `evaluationAsOf` 스냅샷으로 고정하고 발급 직전에 차단·캠페인 종료·운영 중지만 재검증한다. 재시도 한도 소진 뒤 자동 최종 실패로 바꾸지 않고 승인된 `CMD.A.19-22`만 `failed_final`을 확정한다. |
| `HOTSPOT.A.19-07` | 확정 | 판매자는 자신이 소유하거나 비용을 부담한 캠페인의 집계만 조회한다. 사용자·쿠폰 코드·주문 상세는 제외하고 캠페인·드롭·상품·일자 집계, `asOf`, 판매자 부담 비용만 제공한다. |
| `HOTSPOT.A.19-08` | 확정 | 기본적으로 할인 쿠폰 한 장만 허용하고 배송비 쿠폰 한 장은 함께 사용할 수 있다. 그 밖의 조합은 버전이 있는 `stackingPolicyRef`가 명시적으로 허용한 경우에만 적용한다. |
| `HOTSPOT.A.19-09` | 부분 확정 | 생일·생년월일은 자동 지급 조건과 데이터 계약에서 제외한다. 개인정보 원문이 필요 없는 검증된 사건만 허용하며 실제 생산자, Event 유형과 채널은 별도 계약 전까지 활성화하지 않는다. |

## 공통 적용 원칙

- 발급·사용·정책은 결정 시점의 상태와 정책 버전을 변경할 수 없는 원장과 스냅샷으로 남긴다.
- 결제·취소·환불과 승인 결과는 검증 가능한 외부 참조가 있을 때만 반영한다.
- 중복 적용, 위험 작업과 자동 지급은 명시적인 정책·승인·계약이 없으면 거절한다.
- 승인 한도와 Worker 재시도 수치는 버전이 있는 운영 설정으로 관리하며 코드 상수로 고정하지 않는다.

## 남은 외부 계약

`HOTSPOT.A.19-09`에서 개인정보 제외 원칙과 필수 envelope는 확정했다. 다만 생산자별 Event 유형, payload schema, 인증, 전달 채널과 호환성 정책은 원천 Context가 정해지지 않았으므로 아직 배포 계약이 아니다. 임의 생산자나 topic을 만들지 않는다.

## 서비스 구현 후속 작업

비교 기준은 쿠폰 서비스 커밋 `cd604767505c3537db2b44b9692e5f4e545e2981`이다. 이번 작업에서는 구현 코드를 바꾸지 않으며 다음 차이를 후속 구현으로 남긴다.

- `IssueAcceptedResponse.status`를 `accepted`에서 사용자 상태인 `pending`으로 맞추고, 쿠폰함이 수량 확정 전 완료를 표시하지 않는지 검증한다.
- 결제 최종 확정·실패·취소·환불 Event adapter와 결과 불명확 유예를 연결하고, 회수 뒤 새 예약의 유효기간·캠페인·운영 중지 검증을 추가한다.
- 발급·예약의 정책 버전 고정, 기본 중복 적용 규칙과 `stackingPolicyRef` 버전 검증을 반영한다. 현재의 `HOTSPOT.A.19-08 remains open` 오류는 제거한다.
- 위험 기반 승인 정책을 적용해 CS 보상의 `X-Approval-Ref`를 조건부로 검증하고 `caseRef`, 사유, 주문 또는 인시던트 참조를 항상 요구한다.
- 대량 발급의 불변 대상 snapshot과 직전 재검증, 설정 기반 지수 backoff, 승인 전 최종 실패 금지를 Worker 테스트로 보장한다.
- `RM.A.19-03` 판매자 성과 투영과 `API.A.19-16`의 판매자 소유·비용 부담 권한, `groupBy`, `scope`, `asOf`, 자기 부담 금액 응답을 구현한다.
- 자동 지급 route는 생산자별 계약이 확정될 때까지 비활성 상태를 유지하고 생일·생년월일 필드가 어느 입력·저장소에도 들어오지 않는지 검증한다.
