# 쿠폰 및 혜택 요구사항 정의

## 기본 정보

- Requirements ID: `REQ.A.02`
- 프로젝트: DropMong
- 작성 기준일: 2026-07-06
- 주요 사용자: 구매자, 판매자, DropMong 운영자, 마케팅 담당자, CS 담당자, 온콜 담당자
- 주요 이해관계자: 판매자, 플랫폼 운영자, 주문/결제 담당자, 정산 담당자, 메시징/인프라 담당자
- 설계 범위: 쿠폰 발급 주체 관리, 플랫폼 쿠폰 발급, 판매자 쿠폰 발급, 제휴 쿠폰 발급, 선착순 제한, 보유 쿠폰 조회, 주문 적용, 조건 검증, 사용 확정, 사용 취소/회수, 만료, 중복 방지, 비동기 실패 처리, 운영 모니터링
- 제외 범위: 드롭 발견, 상품 상세, 재고 배정, 주문 생성, PG 승인, 배송, 정산 회계, 장기 CRM 자동화

## 구조 분석

### 문서 역할

- `REQ.A.01`은 드롭 발견부터 구매 시도, 주문/결제, 결과 확인까지의 큰 사용자 여정을 다룬다.
- `REQ.A.02`는 그 여정 안에서 쿠폰과 혜택이 맡는 독립 도메인 책임을 다룬다.
- 주문 화면에서 보이는 할인 결과는 `REQ.A.01`에 남기되, 쿠폰 발급 가능 여부, 선착순 수량, 보유 상태, 적용 조건, 사용 확정, 사용 취소, 재처리는 이 문서에서 정의한다.
- 쿠폰은 단순한 UI 할인 표시가 아니라 발급 원장, 사용 원장, 주문 금액 계산, 비동기 처리, 운영 복구가 함께 필요한 업무 객체로 본다.
- 쿠폰 발급 주체는 `플랫폼 운영자`, `판매자`, `플랫폼-판매자 제휴`, `CS/운영 보상`으로 구분하고, 각 쿠폰은 비용 부담 주체와 적용 범위를 가진다.
- 판매자 쿠폰은 판매자의 자기 상품 또는 자기 드롭 범위에서만 적용되며, 플랫폼 정책상 위험한 쿠폰은 운영자 승인 이후 발급된다.
- 플랫폼 운영자는 플랫폼 비용 쿠폰, 제휴 쿠폰, 보상 쿠폰을 발급할 수 있고, 판매자가 발급한 쿠폰의 정책 위반 여부를 검토할 수 있다.

### 근거 자료 구조

- Kurly 글은 주문서에서 보이던 쿠폰이 결제 단계에서 사용할 수 없어진 사례를 통해, 쿠폰 조건 검증이 주문 직전 서버 계산과 다시 만나야 함을 보여준다.
- Oliveyoung 쿠폰 글들은 선착순 발급, Redis 수량 관리, Redis Pub/Sub 한계, RabbitMQ 도입, 대량 발급 독립 처리, MQ 장애, 발급 성공 응답과 실제 발급 불일치를 순서대로 보여준다.
- Oliveyoung 분석 문서는 세일 피크, 쿠폰, 주문, MQ, Redis, Datadog 관측성이 함께 병목이 되는 반복 신호를 정리한다.
- 이 문서는 공개 글의 내부 구현을 그대로 복제하지 않고, DropMong MVP에 필요한 요구사항과 검증 기준으로 변환한다.

## 문제 정의

- 사용자가 겪는 문제: 드롭 오픈 시점에 쿠폰을 받거나 적용했다고 믿었지만, 실제 보유 쿠폰, 주문 금액, 사용 상태가 뒤늦게 달라지면 구매 경험과 신뢰가 크게 떨어진다.
- 현재 방식의 한계: 쿠폰을 주문 화면의 할인 표시로만 다루면 발급 주체, 비용 부담, 판매자 발급 권한, 선착순 발급 수량, 중복 발급, 비동기 발급 실패, 조건 재검증, 사용 취소, 운영 재처리를 설명할 수 없다.
- 이 프로젝트가 해결해야 하는 것: DropMong은 쿠폰 발급 주체와 비용 부담을 명확히 하면서 발급부터 주문 적용과 사용 확정까지 서버 기준으로 일관되게 처리하고, 피크 상황에서도 발급 수량과 사용 상태를 설명 가능하게 남겨야 한다.
- 해결하지 않는 것: 모든 사용자에게 쿠폰 발급 성공을 보장하지 않는다. 외부 메시징, Redis, DB, 주문/결제 시스템 장애 자체를 제거하지 않는다. 대신 장애 영향 범위를 줄이고 추적, 재처리, 회수할 수 있게 한다.

## 목표

- 사용자 목표: 사용자는 쿠폰을 받을 수 있는지, 보유 중인지, 주문에 적용 가능한지, 왜 실패했는지를 납득할 수 있다.
- 비즈니스 목표: DropMong은 플랫폼 쿠폰, 판매자 쿠폰, 제휴 쿠폰으로 구매 전환을 높이면서도 선착순 혜택의 공정성과 신뢰를 지킨다.
- 운영 목표: 운영자는 쿠폰 발급 주체, 비용 부담, 사용, 취소, 실패, 재처리 상태를 쿠폰/드롭/판매자/사용자 단위로 확인하고 조치할 수 있다.
- 기술 목표: 쿠폰 발급과 사용은 동시 요청, 중복 요청, 비동기 실패, Redis/MQ 장애에도 과발급과 중복 사용을 만들지 않는다.

## 제약 조건

- 정책/법적 제약: 쿠폰 유효 기간, 할인 조건, 사용 제한, 비용 부담 주체, 취소/환불 시 회수 기준은 사용자와 판매자에게 필요한 범위로 고지되어야 한다.
- 기술 제약: 피크 순간에는 Redis, MQ, DB 중 하나만 정상이어도 전체 정합성이 자동으로 보장되지 않는다. 각 계층의 역할과 실패 처리를 분리해야 한다.
- 일정 제약: MVP에서는 복잡한 개인화 쿠폰 추천보다 선착순 발급, 보유 조회, 주문 적용, 사용 확정, 회수, 운영 관측을 우선한다.
- 데이터 제약: 주문 금액, 상품 가격, 드롭 상태, 사용자 등급, 판매자 조건은 다른 도메인의 최신 데이터에 의존한다.
- 운영 제약: 수동 DB 수정으로 쿠폰 상태를 바로 바꾸는 운영은 허용하지 않는다. 운영 기능, 감사 로그, 재처리 작업을 통해 변경해야 한다.

## 수용 기준

- 판매 수량이 `N`인 선착순 쿠폰에서 동시 발급 요청 후 실제 발급 수는 `N`을 초과하지 않는다.
- 동일 사용자가 같은 쿠폰을 여러 번 요청해도 보유 쿠폰은 하나만 생성된다.
- 동일 보유 쿠폰을 두 개의 주문에 동시에 적용하려 하면 하나의 주문만 사용 확정된다.
- 사용자가 쿠폰 발급 성공 화면을 본 뒤 Worker가 최종 발급에 실패하면, 운영 재처리 대상과 사용자 상태 조회에 불일치가 남는다.
- 주문서에서 적용 가능하던 쿠폰이라도 결제 직전 서버 재계산에서 조건을 만족하지 못하면 사용 확정되지 않고 사유 코드가 남는다.
- 주문 생성 실패, 결제 실패, 결제 취소, 주문 만료가 발생하면 사용 예약 또는 사용 완료 쿠폰은 정책에 따라 취소/회수된다.
- Redis 장애 또는 MQ 장애가 발생해도 이미 완료된 쿠폰 사용 원장은 손상되지 않고, 신규 발급은 정책에 따라 제한 또는 지연된다.
- DLQ에 들어간 쿠폰 이벤트는 운영자가 원본 요청, 실패 사유, retry count를 확인한 뒤 재처리할 수 있다.
- 운영자는 쿠폰별 발급 요청 수, 성공 응답 수, 실제 발급 수, 미발급 수, 사용 수, 회수 수를 한 화면에서 비교할 수 있다.
- 쿠폰 받기, 보유 조회, 주문 적용, 사용 확정 API의 p95 latency와 error rate가 드롭/쿠폰 단위로 집계된다.
- CS 담당자는 사용자의 쿠폰 발급부터 주문 사용, 취소/회수까지 타임라인으로 확인할 수 있다.
- 판매자 쿠폰은 해당 판매자의 상품, 드롭, 스토어 범위 밖에서는 적용되지 않는다.
- 플랫폼 쿠폰, 판매자 쿠폰, 제휴 쿠폰, 보상 쿠폰은 발급 주체와 비용 부담 주체가 구분되어 기록된다.
- 판매자 쿠폰 또는 제휴 쿠폰이 운영자 승인을 요구하는 정책에 걸리면 승인 전에는 구매자에게 노출되지 않는다.
- 모든 요구사항 ID는 `REQ.A.02.FR-*` 또는 `REQ.A.02.NFR-*` 형식으로 기록된다.

## 새로 드러난 가장자리

- 이 자료가 답한 질문: 쿠폰은 주문 화면의 할인 표시가 아니라 발급 주체, 비용 부담, 발급 수량, 보유 상태, 적용 조건, 사용 원장, 메시징, 운영 복구를 함께 가진 도메인이라는 점을 요구사항으로 분리했다.
- 아직 남은 질문: DropMong이 사용자에게 "발급 접수"와 "발급 완료"를 어떻게 표현할지, 사용 확정 시점을 주문/결제 중 어디에 둘지, 판매자 쿠폰 승인 범위를 어디까지 둘지 아직 정해지지 않았다.
- 검증되지 않은 전제: MVP 쿠폰은 대부분 드롭 단위 선착순 쿠폰이며, 복잡한 개인화 추천이나 다단계 캠페인은 후순위라고 가정했다.
- 약한 연결: 쿠폰 정책 변경, 판매자 쿠폰, 정산, 환불, 보상 지급은 주문/결제/정산/판매자 도메인과 경계가 맞물리므로 후속 문서에서 계약을 더 좁혀야 한다.
- 다음에 확인할 것: Redis/MQ 선택, 쿠폰 상태 전이 모델, 주문 서비스 API 계약, 운영 재처리 권한, 쿠폰 사용 확정과 결제 취소의 보상 규칙, 판매자 쿠폰 승인 정책

- `REQ.A.02.PP-009` 단순 증설로 해결되지 않는 MQ 장애: 이미 동기화가 깨진 상태에서는 인스턴스 증설만으로 복구되지 않고, 관리형 MQ 제약도 있었다. DropMong은 queue type, broker failover, 운영 runbook, 재처리 기준을 요구해야 한다. 근거: [RabbitMQ Classic Queue 메모리 장애와 Quorum Queue 전환기](https://oliveyoung.tech/2025-10-28/coupon-mq-issue/)
- `REQ.A.02.PP-010` 사용자 성공 응답과 실제 발급 실패의 불일치: 사용자에게 성공 응답을 보냈지만 Worker 2차 검증에서 실제 발급이 실패했다. DropMong은 API 성공, MQ 적재, DB 발급 성공, 최종 실패를 분리해 추적해야 한다. 근거: [올영세일 선착순 쿠폰, 미발급 0%를 향한 여정](https://oliveyoung.tech/2025-12-15/fcfs-coupon/)
- `REQ.A.02.PP-011` 비동기 처리의 Time Gap: 1차 유효성 검사와 Worker 2차 검사 사이에 시간 간극이 생겨 앞단 검증 결과가 무의미해질 수 있다. DropMong은 접수 시점 선점 또는 요청 수량/실제 발급 수량 분리 전략을 둬야 한다. 근거: [올영세일 선착순 쿠폰, 미발급 0%를 향한 여정](https://oliveyoung.tech/2025-12-15/fcfs-coupon/)
- `REQ.A.02.PP-012` Redis `GET`/`INCR` 원자성 부재: 개별 Redis 명령은 원자적이지만 조합 로직은 race condition을 만들 수 있다. DropMong은 발급 수량 검증과 차감이 원자적이도록 요구해야 한다. 근거: [올영세일 선착순 쿠폰, 미발급 0%를 향한 여정](https://oliveyoung.tech/2025-12-15/fcfs-coupon/)
- `REQ.A.02.PP-013` 원자성과 성능 사이의 트레이드오프: Lua script로 원자성은 확보했지만 성능 저하가 발생했다. DropMong은 정합성 요구와 함께 latency/throughput 목표를 정의해야 한다. 근거: [올영세일 선착순 쿠폰, 미발급 0%를 향한 여정](https://oliveyoung.tech/2025-12-15/fcfs-coupon/)
- `REQ.A.02.PP-014` 가격/할인 변경으로 쿠폰 조건이 바뀌는 문제: 주문서에서는 조건을 만족했지만 결제 전 상품 할인율 변경으로 쿠폰 조건이 깨졌다. DropMong은 결제 직전 서버 기준 가격 스냅샷으로 쿠폰 조건을 재검증해야 한다. 근거: [Kurly 쿠폰과 할인으로 앞다리살 하나 더 판매한 이야기](https://helloworld.kurly.com/blog/%EC%BF%A0%ED%8F%B0%EA%B3%BC-%ED%95%A0%EC%9D%B8%EC%9C%BC%EB%A1%9C-%EC%95%9E%EB%8B%A4%EB%A6%AC%EC%82%B4-%ED%95%98%EB%82%98-%EB%8D%94-%ED%8C%90%EB%A7%A4%ED%95%9C-%EC%9D%B4%EC%95%BC%EA%B8%B0)
