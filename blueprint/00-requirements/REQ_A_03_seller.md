# 판매자 요구사항 정의

## 기본 정보

- Requirements ID: `REQ.A.03`
- 프로젝트: DropMong
- 작성 기준일: 2026-07-07
- 주요 사용자: 판매자, 판매자 대표 관리자, 판매자 상품 담당자, 판매자 출고 담당자, 판매자 성과 조회자
- 주요 이해관계자: 플랫폼 운영자(DropMong 운영자), CS 담당자, 정산 담당자, 상품/콘텐츠 검수 담당자, 주문/배송 담당자
- 설계 범위: 판매자 등록, 판매자 프로필/스토어 관리, 상품 등록, 드롭 판매 조건 설정, 판매자 쿠폰 등록/관리, 검수 요청, 승인 결과 확인, 드롭 현황 조회, 상품 판매 통계 분석, 주문/출고 자료 조회, 성과 조회, 판매자 내부 권한 관리
- 제외 범위: 플랫폼 전체 운영, 사이트 어드민, CS 운영, 인시던트 선언, 전체 사용자 차단, 상품 제휴/이벤트 제휴 운영, 쿠폰 발급 시스템 내부 처리, PG 승인, 정산 회계 확정

## 구조 분석

### 문서 역할

- `REQ.A.01`은 구매자 관점의 드롭 탐색, 구매 시도, 재고 배정, 주문/결제 경험을 다룬다.
- `REQ.A.02`는 쿠폰과 혜택의 발급, 적용, 사용 확정, 재처리를 다룬다.
- `REQ.A.03`은 DropMong에서 상품을 판매하는 판매자 액터의 요구사항을 다룬다.
- 이 문서에서 사업자 판매자, 전문 셀러, 리셀러는 모두 `판매자`로 통합한다.
- 브랜드명, 스토어명, 프로필, 로고, 소개 문구는 판매자가 자신을 표현하는 선택적 브랜딩 요소로 본다.
- 플랫폼 운영자(DropMong 운영자)는 사이트 운영, CS, 검수 승인, 상품 제휴, 이벤트 제휴, 장애 대응을 담당하는 별도 액터이며 후속 문서에서 다룬다.

### 판매 모델 판단

- KREAM 같은 한정판 플랫폼은 일반 회원 판매, 전문 셀러, 보관 판매, 입점 사업자 판매, 기획전/드롭형 노출이 섞일 수 있다.
- DropMong은 MVP에서도 판매자를 하나의 액터로 통합하되, 판매자 유형은 `사업자`, `전문 셀러`, `리셀러` 같은 속성으로 관리한다.
- 판매자 유형은 인증 수준, 등록 가능 상품, 정산 방식, 노출 제한, 검수 강도에 영향을 줄 수 있지만 요구사항의 기본 주체를 나누지는 않는다.
- 상품 제휴나 이벤트 제휴는 판매자 셀프서비스가 아니라 플랫폼 운영자가 기획하고 승인하는 운영 업무로 분리한다.
- 판매자는 자기 상품 또는 자기 드롭 범위에서 사용할 수 있는 판매자 쿠폰을 발급할 수 있다.
- 플랫폼 운영자가 제휴 쿠폰을 제안할 수 있으며, 판매자는 비용 부담, 적용 범위, 노출 조건을 확인한 뒤 참여 여부를 결정한다.

## 문제 정의

- 사용자가 겪는 문제: 판매자 책임, 상품 보유 여부, 오픈 시간, 판매 수량, 배송/환불 조건이 불명확하면 구매자는 드롭을 신뢰하기 어렵고, 문제가 생겼을 때 누구에게 책임이 있는지 알기 어렵다.
- 현재 방식의 한계: 판매자를 단순 상품 등록자로만 다루면 판매자 유형, 인증 수준, 판매 조건 변경 제한, 판매자 쿠폰 발급, 상품 판매 통계 분석, 주문 정보 접근, 출고 자료 제공, 감사 로그, 개인정보 최소 제공을 설명할 수 없다.
- 이 프로젝트가 해결해야 하는 것: DropMong은 모든 판매자가 같은 판매자 포털에서 상품과 드롭을 준비하게 하되, 판매자 유형에 따른 검증/권한/정책 차이를 시스템이 관리해야 한다.
- 해결하지 않는 것: 상품 제휴와 이벤트 제휴의 플랫폼 운영 절차는 이 문서에서 다루지 않는다. 판매자 정산 회계의 최종 산식과 세금 처리는 별도 정산 문서에서 다룬다.

## 목표

- 사용자 목표: 판매자는 자신의 상품을 드롭으로 등록하고 검수 요청을 제출하며, 승인 결과와 상품 판매 통계를 분석할 수 있다.
- 비즈니스 목표: DropMong은 사업자 판매자와 리셀러를 같은 판매자 모델로 수용하면서도 상품 신뢰와 구매자 보호 기준을 유지한다.
- 운영 목표: 플랫폼 운영자는 판매자가 제출한 상품/드롭을 검수하고, 판매자는 자기 판매 범위 안에서 필요한 정보를 셀프서비스로 확인한다.
- 기술 목표: 판매자 포털은 구매자 구매 경로와 장애 영향을 분리하고, 권한/감사/개인정보 보호 기준을 기본으로 갖춘다.

## 제약 조건

- 정책/법적 제약: 판매자 신원, 판매 책임, 배송/교환/환불 조건, 개인정보 제공 범위는 전자상거래 관련 고지 기준을 충족해야 한다.
- 기술 제약: 판매자 포털은 구매자 구매 경로와 동일한 쓰기 자원에 과도하게 의존하지 않아야 한다.
- 일정 제약: MVP에서는 상품 제휴/이벤트 제휴 운영보다 판매자 등록, 드롭 상품 등록, 검수 요청, 승인 결과 확인, 성과 조회, 주문 자료 조회를 우선한다.
- 데이터 제약: 판매 수량, 옵션, 주문 상태, 출고 상태, 유입 경로, 쿠폰 사용 수, 쿠폰 비용 부담, 취소/환불 요청 수, 판매 통계 지표는 다른 도메인의 최신 데이터에 의존한다.
- 운영 제약: 판매자가 플랫폼 전체 차단, 사용자 보상, 주문 강제 변경, 결제 취소, 상품 제휴/이벤트 제휴 설정을 직접 수행하는 것은 허용하지 않는다.

## 수용 기준

- 판매자는 자기 판매자 계정의 드롭만 조회하고 수정할 수 있다.
- 사업자 판매자, 전문 셀러, 리셀러는 모두 `판매자` 액터로 동작하되 판매자 유형 속성을 가진다.
- 승인되지 않은 드롭은 구매자 홈, 드롭 상세, 검색 결과에 노출되지 않는다.
- 판매자는 드롭 상품, 옵션, 판매 수량, 오픈 시간, 배송/반품 조건을 입력하고 검수 요청을 제출할 수 있다.
- 플랫폼 운영자가 반려한 드롭은 반려 사유가 판매자 화면에 표시된다.
- 승인 후 가격, 판매 수량, 오픈 시각, 구매 제한 변경은 직접 수정되지 않고 변경 요청으로 남는다.
- 진행 중 드롭의 조회 수, 알림 신청 수, 구매 시도 수, 주문 성공 수, 결제 실패 수, 품절 시각이 판매자 화면에서 조회된다.
- 판매자는 상품 판매 통계를 상품, 옵션, 드롭 회차, 기간, 유입 경로, 쿠폰 사용 여부 기준으로 분석할 수 있다.
- 판매자 상품 판매 통계는 지표 정의, 집계 완료 시각, 실시간/사후 확정 여부를 함께 표시한다.
- 판매자는 자기 상품 또는 자기 드롭에만 적용되는 판매자 쿠폰을 등록할 수 있다.
- 판매자는 판매자 쿠폰의 발급 수, 사용 수, 할인 비용, 주문 전환 효과를 확인할 수 있다.
- 판매자는 플랫폼 운영자가 제안한 제휴 쿠폰의 비용 부담과 적용 범위를 확인한 뒤 참여 여부를 응답할 수 있다.
- 판매자 출고 담당자는 자기 판매 주문 목록과 출고에 필요한 최소 정보만 다운로드할 수 있다.
- 주문 자료 다운로드 이력은 요청자, 요청 시각, 범위, 목적과 함께 감사 로그에 남는다.
- 판매자 포털 장애가 발생해도 구매자의 드롭 조회, 구매 시도, 결제 상태 확인은 중단되지 않는다.
- 판매자 귀책 취소, 출고 지연, 재고 부족 같은 이슈는 사유 코드와 상태로 남는다.
- 상품 제휴와 이벤트 제휴는 판매자 기능이 아니라 플랫폼 운영자 기능으로 분리된다.
- 모든 요구사항 ID는 `REQ.A.03.FR-*` 또는 `REQ.A.03.NFR-*` 형식으로 기록된다.

## 열린 질문

- 판매자 유형은 `사업자 판매자`, `전문 셀러`, `리셀러` 세 가지로 시작할 것인가?
- 리셀러도 드롭 오픈 시간을 직접 설정할 수 있는가, 아니면 즉시 판매/판매 입찰만 허용할 것인가?
- 판매자가 드롭을 직접 오픈할 수 있는가, 아니면 플랫폼 운영자 승인 후 자동 오픈만 가능한가?
- 승인 후 핵심 조건 변경은 어떤 시간 전까지 허용할 것인가?
- 판매자가 입력한 판매 수량은 실제 보유/공급 증빙과 어떻게 연결할 것인가?
- 주문 자료 다운로드에서 판매자에게 제공할 개인정보 최소 범위는 어디까지인가?
- 판매자 귀책 취소 또는 출고 지연 시 구매자 보상 권한은 판매자와 플랫폼 운영자 중 어디에 둘 것인가?
- 판매자 쿠폰은 판매자가 즉시 발급할 수 있는가, 아니면 운영자 승인 후 활성화할 것인가?
- 플랫폼-판매자 제휴 쿠폰의 비용 부담 비율은 판매자가 직접 승인해야 하는가?

## 확인 필요

- 판매자 등록에 필요한 신원 정보, 사업자 정보, 본인 인증 정보, 담당자 정보
- 판매자 유형별 상품 등록 가능 범위와 검수 강도
- 배송 주체: 판매자 직접 출고, DropMong 위탁 출고, 외부 3PL 연동 여부
- 드롭 검수 기준: 상품 이미지, 가격, 재고, 지식재산권, 배송/환불 고지, 금지 상품
- 승인 후 변경 가능 범위와 변경 요청 SLA
- 판매자 성과 지표 정의: 조회, 알림 신청, 구매 시도, 재고 배정, 주문 성공, 품절 시각, 옵션별 판매량, 유입 경로, 쿠폰 사용 효과, 취소/환불률
- 판매자 쿠폰 정책: 허용 할인율, 발급 수량 상한, 적용 가능 상품/드롭 범위, 운영자 승인 필요 조건
- 주문 자료 다운로드 포맷과 개인정보 마스킹 기준
- 판매자 귀책 사유 코드와 정산/보상 연계 기준

## 새로 드러난 가장자리

- 이 자료가 답한 질문: DropMong에는 구매자와 플랫폼 운영자만 있는 것이 아니라, 드롭 상품을 공급하고 판매 조건을 준비하는 판매자 액터가 필요하다.
- 아직 남은 질문: 사업자 판매자, 전문 셀러, 리셀러의 정책 차이를 어디까지 MVP에 포함할지 아직 정해지지 않았다.
- 검증되지 않은 전제: DropMong은 판매자를 하나의 액터로 통합하고, 판매자 유형은 인증/검수/정산/노출 정책의 속성으로 관리한다고 가정했다.
- 약한 연결: 정산, 출고, 반품, 판매자 귀책 보상, 판매자 쿠폰, 상품 판매 통계 분석은 `REQ.A.03`만으로 닫히지 않고 주문/배송/정산/쿠폰/분석 문서와 연결되어야 한다.
- 다음에 확인할 것: 판매자 유형 정책, 배송 주체, 검수 승인 SLA, 개인정보 제공 범위, 정산 차감 기준, 판매자 귀책 보상 정책, 판매 통계 집계 기준, 판매자 쿠폰 승인 정책
