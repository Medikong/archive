# 플랫폼 운영자 사이트 요구사항 정의

## 기본 정보

- Requirements ID: `REQ.A.04`
- 프로젝트: DropMong
- 작성 기준일: 2026-07-07
- 주요 사용자: 플랫폼 운영자, CS 담당자, 상품/콘텐츠 검수 담당자, 제휴/이벤트 담당자, 정산 담당자, 운영 관리자, 인시던트 담당자
- 주요 이해관계자: 구매자, 판매자, DropMong 운영 조직, 개발/SRE 담당자, 정산/회계 담당자, 법무/정책 담당자
- 설계 범위: 운영자 로그인, 운영자 권한 관리, 판매자 심사, 상품/드롭 검수, 드롭 운영 현황, 플랫폼 쿠폰 발급, 판매자 쿠폰 승인, 제휴 쿠폰 제안, CS 조회/처리, 제휴/이벤트 운영, 위험 작업 승인, 운영 공지, 인시던트 대응, 감사 로그, 운영 지표 대시보드
- 제외 범위: 구매자 홈/상세/주문 UI, 판매자 포털 셀프서비스, 결제 승인 처리, 물류 실행 시스템, 최종 회계 처리, 외부 제휴 계약서 관리

## 구조 분석

### 문서 역할

- `REQ.A.01`은 구매자 관점의 드롭 탐색, 구매 시도, 재고 배정, 주문/결제 경험을 다룬다.
- `REQ.A.02`는 쿠폰과 혜택의 발급, 적용, 사용 확정, 재처리를 다룬다.
- `REQ.A.03`은 사업자 판매자, 전문 셀러, 리셀러를 모두 포괄하는 판매자 요구사항을 다룬다.
- `REQ.A.04`는 DropMong 내부 운영자가 사용하는 플랫폼 운영자 사이트 요구사항을 다룬다.
- 이 문서에서 `운영자`는 판매자가 아니라 DropMong 플랫폼 운영 조직을 의미한다.
- 플랫폼 운영자는 CS, 사이트 운영, 상품/드롭 검수, 판매자 심사, 플랫폼 쿠폰 발급, 판매자 쿠폰 승인, 상품 제휴, 이벤트 제휴, 장애 대응, 위험 작업 승인에 책임을 가진다.

### 운영자 사이트 판단

- DropMong의 운영자 사이트는 단순 CRUD 어드민이 아니라 피크 드롭 순간에 서비스 신뢰를 지키는 운영 콘솔이다.
- 판매자는 자기 상품과 주문 범위만 다루고, 플랫폼 운영자는 판매자·상품·드롭·주문·쿠폰·CS·공지·인시던트 상태를 횡단해서 확인한다.
- 상품 제휴와 이벤트 제휴는 판매자의 셀프서비스가 아니라 플랫폼 운영자가 기획, 승인, 노출, 운영 상태를 관리하는 업무로 둔다.
- 플랫폼 운영자는 플랫폼 비용 쿠폰, 플랫폼-판매자 제휴 쿠폰, CS/운영 보상 쿠폰을 발급할 수 있다.
- 플랫폼 운영자는 판매자가 등록한 쿠폰 중 정책 검토가 필요한 쿠폰을 승인, 반려, 보류 처리한다.
- 운영자 사이트는 구매자 구매 경로보다 높은 권한을 가지므로 RBAC, 감사 로그, 이중 확인, 위험 작업 제한이 기본 요구사항이다.
- 운영자 사이트 장애가 구매자 드롭 참여와 결제 경로에 직접 영향을 주지 않도록 서비스와 조회 부하를 분리해야 한다.

## 문제 정의

- 사용자가 겪는 문제: 구매자와 판매자는 드롭 오픈 중 문제가 생겼을 때 빠른 안내, 정확한 상태 확인, 일관된 보상/취소/공지 처리를 기대한다.
- 현재 방식의 한계: 운영자가 여러 DB, 로그, 대시보드, 채팅, 스프레드시트를 오가며 처리하면 피크 시간에 판단이 늦어지고, 처리 기준이 담당자마다 달라진다.
- 이 프로젝트가 해결해야 하는 것: DropMong은 운영자가 판매자, 상품, 드롭, 주문, 쿠폰, CS, 인시던트 상태를 한 화면 체계에서 확인하고 안전하게 조치할 수 있는 운영자 사이트를 제공해야 한다.
- 해결하지 않는 것: 모든 운영 자동화를 MVP에서 완성하지 않는다. 결제 승인, 배송 실행, 회계 확정, 법무 계약 관리는 각 전문 시스템 또는 별도 문서에서 다룬다.

## 목표

- 사용자 목표: 운영자는 드롭 진행 상태와 문제 상황을 빠르게 파악하고, 구매자와 판매자에게 일관된 기준으로 안내하거나 조치할 수 있다.
- 비즈니스 목표: DropMong은 한정 드롭의 공정성, 신뢰, 제휴 운영 품질을 유지하면서 운영 인력의 반복 업무를 줄인다.
- 운영 목표: CS, 검수, 제휴, 정산, 인시던트 담당자가 같은 운영 데이터와 정책 기준을 보고 협업한다.
- 기술 목표: 운영자 사이트는 높은 권한과 넓은 조회 범위를 가지되, 구매자 구매 경로와 장애 영향을 분리하고 모든 위험 작업을 감사 가능하게 만든다.

- 정책/법적 제약: 개인정보 조회, 주문 처리, 고객 안내, 판매자 책임 고지, 전자상거래 분쟁 대응 기준을 지켜야 한다.
- 기술 제약: 운영자 사이트는 여러 도메인 서비스를 조회하므로 API 응답 지연, stale data, 장애 격리를 전제로 설계해야 한다.
- 일정 제약: MVP에서는 운영자 권한, 판매자/상품 검수, 드롭 현황, CS 조회, 공지, 인시던트, 감사 로그를 우선한다.
- 데이터 제약: 주문, 쿠폰, 결제, 재고, 알림, 정산 데이터는 서로 다른 서비스의 상태를 조합하므로 출처와 최신성이 필요하다.
- 운영 제약: 운영자는 강한 권한을 가지지만, 구매자 공정성을 해치는 임의 조작이나 근거 없는 보상/취소 처리는 허용하지 않는다.

## 수용 기준

- 운영자는 역할에 따라 허용된 메뉴와 작업만 수행할 수 있다.
- 운영자는 판매자 신청을 승인, 반려, 보류 처리하고 구조화된 사유를 남길 수 있다.
- 운영자는 판매자가 제출한 상품과 드롭 조건을 검수하고 승인 또는 반려할 수 있다.
- 승인 후 핵심 드롭 조건 변경은 변경 요청과 위험 작업 승인 절차를 거친다.
- CS 담당자는 주문 상세에서 주문, 결제, 쿠폰, 재고 배정, 배송 상태와 최신성을 함께 확인할 수 있다.
- 운영자는 진행 중인 드롭의 주요 지표와 이상 신호를 확인할 수 있다.
- 운영자는 상품 제휴와 이벤트 제휴 드롭을 일반 드롭과 구분해 관리할 수 있다.
- 운영자는 플랫폼 비용 쿠폰과 CS/운영 보상 쿠폰을 발급할 수 있다.
- 운영자는 판매자가 등록한 쿠폰을 승인, 반려, 보류 처리할 수 있다.
- 운영자는 플랫폼-판매자 제휴 쿠폰을 제안하고 판매자 응답 상태와 비용 부담 기준을 확인할 수 있다.
- 인시던트 담당자는 장애를 선언하고 심각도, 영향 범위, 조치 이력, 고객 안내 상태를 남길 수 있다.
- 운영자는 드롭 공지, 장애 안내, 보상 안내 문구를 등록하고 노출 상태를 관리할 수 있다.
- 재처리와 보정 요청에는 고유 요청 ID, 멱등키, 요청자, 승인자, 결과 상태가 남는다.
- 모든 운영자 위험 작업은 감사 로그로 재구성 가능하다.
- 운영자 사이트 장애가 구매자의 드롭 탐색, 구매 시도, 결제 상태 확인을 중단시키지 않는다.
- 모든 요구사항 ID는 `REQ.A.04.FR-*` 또는 `REQ.A.04.NFR-*` 형식으로 기록된다.

- 운영자 사이트에서 조회 가능한 개인정보 범위와 마스킹 기준은 어떻게 정할 것인가?

## 새로 드러난 가장자리

- 이 자료가 답한 질문: DropMong의 `운영자`는 판매자가 아니라 플랫폼 운영 조직이며, CS·검수·제휴·인시던트·정산 업무를 다루는 운영자 사이트가 필요하다.
- 아직 남은 질문: 운영자에게 실제 상태 변경 권한을 어디까지 줄지, 요청 등록만 허용할지 정해야 한다.
- 검증되지 않은 전제: MVP에서도 운영자 사이트는 단순 관리자 CRUD가 아니라 피크 드롭 운영 콘솔로 시작한다고 가정했다.
- 약한 연결: 운영자 사이트의 쿠폰 발급, 보정/재처리 기능은 주문, 쿠폰, 결제, 정산 서비스의 실제 API 정책과 연결되어야 한다.
- 다음에 확인할 것: 운영자 권한 매트릭스, 위험 작업 승인선, CS 개인정보 범위, 인시던트 심각도 기준, 제휴/이벤트 운영 범위, 쿠폰 발급 승인 정책
