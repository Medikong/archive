## 역할과 도메인 책임 원칙

2026-07-13 데일리 미팅에서 정한 역할 A, B, C는 프로젝트 전체에 적용할 기반 구조의 주 담당을 뜻한다. 이번 구현에서는 이 역할과 별개로 각 구성원이 자신이 맡은 도메인을 처음부터 끝까지 책임진다.

도메인 책임에는 다음 작업이 포함된다.

- 사용자 문제와 도메인 규칙 정의
- 도메인 모델과 상태 전이 설계
- API 및 이벤트 계약 작성
- 애플리케이션 구현
- 데이터 저장 방식과 장애 대응 설계
- 기능, 통합, 실패 시나리오 테스트
- 배포 구성과 관측성 적용
- 설계 근거, 검증 결과, 남은 위험 문서화

구현 결과만 만드는 데서 끝내지 않는다. 발생 가능한 문제를 먼저 정의하고, 이를 막거나 복구하는 방법까지 코드와 검증 결과로 보여주는 것을 공통 기준으로 삼는다.

## 최범휘 피드백

### 받은 피드백

- 가용 영역을 역할별로 나누는 구성보다 A 존과 B 존 두 곳을 사용해 이중화를 검증하는 편이 더 실무적일 수 있다.
- 역할별 분리를 위해 존을 늘리는 구성이 실제로 효율적인지 다시 따져봐야 한다.
- 결제 기능은 구현했다는 설명만으로 부족하다. 실제 성공과 실패 상황에서 의도한 대로 동작하는지 보여줘야 한다.
- 기술 선택과 구현 결과를 포트폴리오에 자연스럽게 녹여내려면 본인이 문제를 정의하고 판단한 근거가 드러나야 한다.

### 설계 및 구현 과제

| 주제 | 확인할 내용 | 필요한 증거 |
| --- | --- | --- |
| A/B 존 이중화 | 어떤 컴포넌트를 두 존에 배치하고, 한 존이 중단됐을 때 트래픽과 상태를 어떻게 유지할지 정한다. | 배치 구성도, 장애 시나리오, 전환 결과 |
| 존 분리 기준 | 역할, 서비스, 데이터 중요도 중 무엇을 기준으로 존을 나누는지 설명한다. | 대안 비교와 선택 근거 |
| 결제 검증 | 승인 성공뿐 아니라 실패, 시간 초과, 중복 요청, 결과 확인 지연을 다룬다. | 테스트 결과, 로그와 지표, 복구 절차 |
| 포트폴리오 정리 | 문제, 선택한 방법, 포기한 대안, 검증 결과, 배운 점을 본인의 판단으로 설명한다. | 의사결정 기록과 재현 가능한 결과 |

### 확인할 질문

- 두 존을 사용하는 목적은 서비스 가용성 검증인가, Kubernetes 스케줄링 검증인가?
- 한 존이 중단되면 어떤 기능까지 계속 제공해야 하는가?
- 결제 성공 응답을 받지 못했지만 실제 승인이 완료된 경우 상태를 어떻게 맞출 것인가?
- 이중화로 얻는 이점이 추가 비용과 운영 복잡도보다 큰가?

### 보완이 필요한 원문

`PO와 PM`에 관한 메모는 논의 내용이 완결된 문장으로 남아 있지 않다. 제품 목표, 우선순위, 이해관계자 조율 중 어떤 주제였는지 확인한 뒤 구체적인 피드백을 보완한다.

## 김정엽 피드백

### 받은 피드백

- 드롭 서비스의 특성을 고려해 다른 서비스에 지나치게 의존하지 않는 랭킹 방식을 검토한다.
- 사용자가 찜한 상품과 상품별 관심 신호를 랭킹 입력값으로 활용할 수 있다.
- 누적 조회수를 그대로 사용하면 판매 기간이 긴 상품이 유리해지는 편향이 생긴다.
- 원가 대비 할인율이 제공된다면 할인율을 랭킹 신호로 사용하는 방안도 비교해 볼 수 있다.
- 현재 구상은 3시간마다 조회수를 집계해 랭킹을 갱신하는 방식이다. 이를 실시간 랭킹이라고 부를 수 있는지, 상품과 사용자에게 어떤 이점이 있는지 설명해야 한다.
- 같은 사용자가 한 상품을 100번 조회하더라도 랭킹에는 한 번만 반영되도록 `user_id`를 기준으로 중복을 막아야 한다.

### 설계 및 구현 과제

| 주제 | 확인할 내용 | 필요한 증거 |
| --- | --- | --- |
| 랭킹 목표 | 인기 상품 노출, 구매 전환, 신규 상품 발견 중 무엇을 우선할지 정한다. | 목표 지표와 사용자 시나리오 |
| 점수 산정 | 조회, 찜, 할인율을 각각 사용할 때의 장단점과 가중치를 비교한다. | 점수식, 예시 데이터, 결과 비교 |
| 기간 편향 | 오래 노출된 상품이 누적 집계에서 유리해지는 문제를 시간 구간, 감쇠, 정규화로 보정한다. | 보정 전후 순위 비교 |
| 유일 조회 | 사용자와 상품의 조합을 어떤 기간 안에서 한 번으로 볼지 정한다. | 중복 방지 키, 집계 기간, 중복 요청 테스트 |
| 갱신 주기 | 3시간 배치가 필요한 이유와 사용자가 체감하는 최신성 기준을 정의한다. | 갱신 지연, 처리 비용, 장애 시 동작 |
| 서비스 의존성 | 상품 가격이나 할인율을 사용할 경우 데이터 소유 서비스와 전달 방식을 정한다. | API 또는 이벤트 계약, 데이터 지연 허용 범위 |

### 확인할 질문

- `실시간`은 이벤트가 들어오는 즉시 반영한다는 뜻인가, 3시간 이내의 최신 순위를 제공한다는 뜻인가?
- 유일 조회의 집계 기간은 3시간, 하루, 상품 판매 기간 중 어느 범위인가?
- 로그인하지 않은 사용자의 조회는 제외할지, 별도 식별자로 제한할지 결정했는가?
- 할인율이 높은 상품만 상위에 고정되는 부작용은 어떻게 막을 것인가?
- 랭킹이 실제로 구매 전환이나 상품 발견에 도움이 됐는지 어떤 지표로 확인할 것인가?

## 박명수 피드백

### 받은 피드백

- 사용자가 환불을 요청한 뒤 어떤 순서로 처리되는지가 중요하다.
- 반품 배송비를 착불로 처리할지, 오배송처럼 판매자 책임이 있는 경우에는 어떻게 처리할지 기준이 필요하다.
- 환불을 결제 도메인 내부에서 처리할지 별도 책임으로 분리할지도 검토해야 한다.
- 결제 도메인에서는 데이터베이스의 정합성과 이중화 구성이 중요하다.
- Python과 Go를 함께 사용하고 있으므로 서비스 간 통신을 REST API로 유지할지 gRPC를 사용할지 비교해 볼 수 있다.
- Kubernetes 생태계와의 연계, 성능, 확장성을 고려하면 Go가 유리할 수 있다. 다만 애플리케이션 기능을 완성하기 위해 Python을 사용하는 것 자체는 문제가 아니다.
- 정상 결과만 만드는 개발보다 발생 가능한 이슈를 정의하고 방어 방법까지 구현하는 개발이 한 단계 더 나아간 접근이다.

### 설계 및 구현 과제

| 주제 | 확인할 내용 | 필요한 증거 |
| --- | --- | --- |
| 환불 절차 | 요청, 승인, 반품, 결제 취소, 완료까지의 상태와 책임자를 정한다. | 상태 전이도, 실패 및 재처리 규칙 |
| 귀책과 배송비 | 단순 변심, 오배송, 상품 하자에 따라 배송비 부담과 승인 조건을 구분한다. | 정책 표와 테스트 시나리오 |
| 서비스 경계 | 환불이 결제, 주문, 별도 환불 책임 중 어디에 속하는지 정한다. | 도메인 경계와 API 또는 이벤트 계약 |
| 결제 정합성 | 중복 취소, 부분 환불, 외부 결제 결과 지연, 재시도를 안전하게 처리한다. | 멱등성 키, 거래 원장, 대사 절차 |
| 데이터베이스 이중화 | 허용할 데이터 손실과 복구 시간을 먼저 정하고 복제 및 장애 전환 방식을 선택한다. | RPO/RTO, 전환 실험, 복구 절차 |
| 서비스 간 통신 | REST와 gRPC를 언어 취향이 아니라 호출 특성, 계약 관리, 운영 난이도로 비교한다. | 간단한 성능 측정과 선택 근거 |

### 확인할 질문

- 환불 완료의 기준은 내부 상태 변경인가, 결제 대행사의 취소 확정인가?
- 주문과 결제의 상태가 어긋나면 어느 데이터를 기준으로 복구할 것인가?
- 부분 환불과 여러 번의 환불 요청을 어떻게 제한할 것인가?
- 주 데이터베이스 장애 중 들어온 요청을 거부할지 대기시킬지 정했는가?
- gRPC로 바꿨을 때 얻는 이점이 계약 관리와 운영 복잡도를 감수할 만큼 큰가?

## 공통 완료 기준

각 구성원은 담당 도메인에서 다음 결과를 남긴다.

1. 해결하려는 사용자 문제와 도메인 경계를 문서로 설명한다.
2. 핵심 규칙, 상태 전이, 데이터 소유권을 설계한다.
3. API 또는 이벤트 계약을 작성하고 실제 코드에 연결한다.
4. 정상 상황과 실패 상황을 모두 구현하고 테스트한다.
5. 배포 후 확인할 로그, 메트릭, 추적 정보를 정한다.
6. 설계 대안과 선택 근거, 포기한 범위를 기록한다.
7. 검증 결과를 재현할 수 있는 실행 방법과 증거를 남긴다.
