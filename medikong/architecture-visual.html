DropMong은 정해진 시간에 한정 수량 상품을 판매하고, 주문 폭주에도 재고 초과 확정을 막는 드롭 커머스 서비스다.

00. 서비스 개요

DropMong은 드롭 피크를 견디는 한정 수량 커머스다

상품을 보여주는 것보다 중요한 기준은 주문 정합성, 안전한 재시도, 장애 격리다.

고객 경험

고객은 오픈 예정 드롭을 보고, 오픈하면 주문하고 결제한다. 같은 요청을 다시 보내도 중복 주문은 없다.

운영자 경험

운영자는 상품, 드롭 일정, 판매 수량을 설정하고, 피크 트래픽과 장애 상태를 dashboard와 runbook으로 확인한다.

시스템 기준

order-service가 재고 예약과 주문 상태를 단독으로 갖는다. 결제와 알림은 이벤트로 연결하고, 실패 시에도 복구 가능한 흐름으로 처리한다.

01. 사용자 흐름

고객 구매 흐름은 6단계다

운영자와 플랫폼 운영자 흐름은 12-user-flows.md에 정리했다.

01 드롭 발견 목록과 상세에서 오픈 시간, 상품, 가격을 확인한다.
02 로그인 주문 전에 JWT를 받고 고객 권한을 확인한다.
03 오픈 대기 오픈 시각까지 상세 화면에서 상태를 확인한다.
04 주문 시도 입장 제어를 통과하면 주문 서비스가 재고를 예약한다.
05 결제 결제 서비스가 승인, 실패, 지연 결과를 이벤트로 남긴다.
06 상태 확인 확정, 취소, 만료 상태와 알림을 확인한다.
02. 설계 문서 세트

구현 전에 합의할 문서 순서

문서는 제품 범위에서 시작해 계약, 운영, 릴리즈 게이트로 끝난다.

읽는 순서
  1. 제품과 도메인 경계를 먼저 잠근다.
  2. 서비스, 데이터, API, 이벤트 계약을 맞춘다.
  3. 핵심 흐름과 인프라 운영 기준을 연결한다.
  4. 사용자 행동과 화면 흐름을 맞춘다.
  5. 보안과 릴리즈 게이트로 구현 착수 조건을 닫는다.

03. 서비스 경계

5개 서비스로 충분하다

5개 서비스 oversell 0 order가 inventory 소유

왜 `order-service`가 재고를 갖나

주문 생성과 재고 예약이 분리되면 동시성 실패와 보상 흐름이 복잡해진다. oversell 0은 같은 트랜잭션 경계에서 지켜야 한다.

왜 `ticket-service`를 빼나

드롭 커머스에서 티켓 발행은 핵심 도메인이 아니다. 주문 확정 결과나 주문 이력으로 흡수하는 편이 경계가 단순하다.

왜 `catalog-service`는 캐시 가능한가

상품과 드롭 정보는 low-mutation read path다. 공개 조회는 projection과 cache로 빠르게 만들고, 재고 진실은 order DB에 둔다.

04. 드롭 오픈 흐름

피크 트래픽은 주문 전에 거른다

대기열, rate limit, admission token은 `order-service` 앞에서 작동해야 한다.

06. 인프라 제어

Kong 제거 후 Istio 중심으로 정리한다

Gateway, traffic split, canary, telemetry가 한 경로에서 설명되어야 한다.

계층 채택 중요한 이유 주요 신호
진입 Istio Ingress Gateway Kong route와 mesh route를 이중으로 설명하지 않는다. accepted_rps / rejected_rps
확장 HPA + KEDA HTTP 서비스는 HPA, Kafka consumer는 lag 기반 KEDA가 맞다. consumer_lag / hpa_replicas
릴리즈 Argo Rollouts order/payment 경로는 canary 분석과 자동 rollback이 필요하다. canary_error_rate / rollback_time
관측성 Prometheus, Grafana, Loki, Tempo metric, log, trace를 correlation id로 연결한다. trace_id / request_id
