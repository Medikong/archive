# 찜(위시리스트) API 설계 비교

## 질문

찜 등록/취소 API는 엔드포인트와 HTTP 메서드를 어떻게 설계하고, 중복 요청(같은 찜을 두 번 누름)에 대한 멱등성은 어떻게 보장하는가?

## 원문에서 확인한 내용

[좋아요 멱등성: 올리브영, 무신사, 컬리는 어떻게 다르게 구현했을까?](https://beta.velog.io/@sweet_sumin/%EC%A2%8B%EC%95%84%EC%9A%94-%EB%A9%B1%EB%93%B1%EC%84%B1-%EC%98%AC%EB%A6%AC%EB%B8%8C%EC%98%81-%EB%AC%B4%EC%8B%A0%EC%82%AC-%EC%BB%AC%EB%A6%AC%EB%8A%94-%EC%96%B4%EB%96%BB%EA%B2%8C-%EB%8B%A4%EB%A5%B4%EA%B2%8C-%EA%B5%AC%ED%98%84%ED%96%88%EC%9D%84%EA%B9%8C)는 실제 요청을 관찰해 세 회사의 찜 API 설계를 비교한다.

- **올리브영**: 등록/취소를 각각 별도 POST 엔드포인트로 분리(`getWishListJson.do` / `delWishLstAjax.do`). 액션이 URL에 명시적으로 드러나 이해하기 쉽지만, 엔드포인트가 늘어난다. 멱등성은 DB 고유 제약 조건 + 트랜잭션/락으로 보장.
- **무신사**: 단일 POST 엔드포인트(`/log/user-event/v2_cart_purchase_like`)에 `event_name`(`add_to_wishlist`/`remove_to_wishlist`)으로 액션을 구분. 클라이언트가 하나의 URL만 알면 되지만, "액션이 body 안에 숨어 있어" REST 관점에서는 덜 명시적이다. 이미 존재하면 중복 추가를 막고, 없으면 성공 응답만 반환하는 방식으로 멱등성을 처리.
- **컬리**: PUT 메서드로 자원의 "최종 상태"를 통째로 지정(`PUT /picks/products/{productId}` + body `{"is_pick": true/false}`). PUT은 정의상 멱등하므로, 같은 요청을 여러 번 보내도 서버 상태가 같아 별도의 멱등성 처리 로직이 단순해진다.

## DropMong 적용 아이디어

- interest-service의 찜 API는 컬리 방식(`PUT /interests/drops/{dropId}` + `{"wished": true/false}`)을 우선 검토할 만하다. HTTP 메서드 자체가 멱등성을 보장해 서버 구현이 단순해지고, 프론트엔드에서 더블 클릭/네트워크 재시도가 발생해도 안전하다.
- 다만 "찜 등록"과 "찜 취소"를 이벤트로 구분해 로깅/집계해야 하는 경우(예: 오픈 전 랭킹 점수 계산에 "누가 언제 찜했는지"가 필요하면) 무신사처럼 액션 기반 이벤트 로그를 별도로 남기는 방식과 결합할 수 있다. 즉 "현재 상태"는 PUT으로 멱등하게 관리하고, "상태 변화 이벤트"는 내부적으로 별도 로그/이벤트로 남기는 두 계층 분리가 가능하다.
- 멱등성 보장의 최종 수단은 결국 DB 고유 제약(`UNIQUE(user_id, drop_id)`)이라는 점은 세 회사 모두 공통이다. interest-service도 API 설계와 무관하게 이 제약을 기본으로 둔다.

## 확인 필요

- 원문은 세 회사의 실제 네트워크 요청을 관찰해 정리한 개인 기술 블로그로, 각 회사의 내부 동시성 처리(락 종류, 트랜잭션 격리 수준)까지 공식적으로 확인된 내용은 아니다.
