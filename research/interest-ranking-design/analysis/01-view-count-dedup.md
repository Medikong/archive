# 조회수 중복 집계 방지 패턴

## 질문

동일 사용자의 반복 조회가 조회수를 부풀리지 않게 하려면 어떻게 구현하는가?

## 원문에서 확인한 내용

- [동일한 유저의 조회수가 중복 집계 되지 않게 하려면?](https://velog.io/@horang12/%EB%8F%99%EC%9D%BC%ED%95%9C-%EC%9C%A0%EC%A0%80%EC%9D%98-%EC%A1%B0%ED%9A%8C%EC%88%98%EA%B0%80-%EC%A4%91%EB%B3%B5-%EC%A7%91%EA%B3%84-%EB%90%98%EC%A7%80-%EC%95%8A%EA%B2%8C-%ED%95%98%EB%A0%A4%EB%A9%B4)은 Redis `SETNX` 명령과 TTL을 사용해, 특정 사용자가 일정 시간 동안 특정 리소스에 대해 조회수를 한 번만 증가시키는 방식을 설명한다. 키가 최초 접속 시에만 설정되고, TTL이 지나기 전까지는 동일 키 생성이 방지된다.
- 같은 글은 "visit" 키를 가진 List에 리소스 ID가 있는지 확인해, 없으면 리스트에 추가하고 조회수를 증가시키고, 있으면 조회수를 올리지 않는 방식도 함께 설명한다. 조회수는 동일 사용자에 대해 하루에 한 번씩 카운팅되도록 구현된 사례가 있다.
- [[재능교환소] 조회수 중복 방지 (Redis, 쿠키)](https://velog.io/@10000ji_/%EC%9E%AC%EB%8A%A5%EA%B5%90%ED%99%98%EC%86%8C-%EC%A1%B0%ED%9A%8C%EC%88%98-%EC%A4%91%EB%B3%B5-%EB%B0%A9%EC%A7%80-Redis-%EC%BF%A0%ED%82%A4)는 비로그인 유저는 쿠키로, 로그인 유저는 Redis로 중복 방지를 분리해서 설계하는 패턴을 제시한다. 사용자 고유 식별자를 키로 쓰고, 값으로 조회한 리소스 ID 목록을 저장한다.

## DropMong 적용 아이디어

- interest-service는 `SET NX EX 300 "viewed:{dropId}:{userId}"` 키로 5분 dedup을 적용한다 (`REQ.A.07.FR-004`).
- 위 글들은 로그인/비로그인을 분리하는 패턴도 보여주지만, DropMong은 1차 스코프에서 **로그인 사용자만** 조회수를 집계하기로 결정했다 (`REQ.A.07.FR-007`). 비로그인 쿠키 기반 dedup은 조작이 쉬워(쿠키 초기화로 우회 가능) 매크로 방어력이 약하다는 게 이 결정의 근거다. 로그인 계정은 auth-service 가입 절차를 거쳐야 해서 매크로 비용이 더 크다.
- TTL 값(위 사례는 하루 단위 예시도 있음)은 도메인마다 다르게 가져갈 수 있다. DropMong은 "짧은 시간 내 새로고침 방지"가 목적이라 5분으로 정했고, 실측 후 조정 여지를 열어둔다 (`REQ.A.07` 열린 질문).

## 확인 필요

- 두 출처 모두 개인 개발 블로그로, 실제 프로덕션 규모에서의 성능/정확도 검증 데이터는 없다. 대용량 트래픽에서의 Redis 부하는 DropMong 자체 부하테스트로 확인해야 한다.
