# 조회/관심 측정 방식 비교

## 질문

"조회했다" 또는 "관심 있다"를 어떤 기준으로 정의하고 측정하는가? (01번 문서가 "중복을 어떻게 막는가"라면, 이 문서는 "애초에 무엇을 조회/관심으로 볼 것인가"를 다룬다.)

## 원문에서 확인한 내용

- [YouTube](https://uppbeat.io/blog/youtube/how-does-youtube-count-views)는 "조회"를 단순 클릭이 아니라 **30초 이상 시청**(20초 미만 영상은 완주)으로 정의한다. 즉 클릭 자체가 아니라 일정 시간 이상의 관여를 조건으로 삼는다.
- YouTube는 조회를 "우선 집계 → 사후 품질 검사 → 행동 패턴 검증 → 최종 조정"의 다단계로 처리하며, 조회수 300 도달 시점에 한 번 더 검증(freeze)한다.
- [무신사](https://www.musinsa.com/content/mz/6622), [크림](https://kream.co.kr/content/ranking_all-rising)은 "조회"를 단일 신호로 쓰지 않고, 구매/거래/관심(찜) 등 **여러 신호를 조합**해 인기도를 정의한다.
- [네이버/다음](https://www.clien.net/service/board/park/5599879)은 "조회"보다 "조회의 증가 속도"(최근 평균 대비 비율)를 급상승 판정 기준으로 쓴다.

## DropMong 적용 아이디어

- 현재 DropMong의 조회수는 "상세 페이지 진입 = 조회"로 단순 정의하고 있다 (`REQ.A.07.FR-004`). YouTube처럼 "일정 시간 이상 페이지에 머물러야 조회로 인정"하는 체류시간 기준을 도입하면 어뷰징 저항력이 높아지지만, 프론트엔드에서 체류시간을 측정해 서버로 보내야 해서 구현 복잡도가 늘어난다. 1차 스코프에서는 "진입 = 조회"로 단순하게 가되, 이 트레이드오프를 `REQ.A.07` 확인 필요 항목에 남긴다.
- 무신사/크림처럼 여러 신호를 조합하는 방식은 DropMong이 이미 채택한 "찜+조회수" 조합과 같은 방향이다. 향후 신호를 늘릴 때(예: 공유, 알림신청)도 이 원칙(단일 신호에 의존하지 않는다)을 유지한다.
- 네이버/다음의 "증가 속도" 개념은 오픈 후 랭킹의 `elapsed_minutes` 정규화와 본질적으로 같은 문제의식(절대량보다 상대적 변화 속도)이다. `02-realtime-ranking-score-design.md`와 함께 참고한다.

## 확인 필요

- YouTube의 "체류시간 기준 조회 정의"를 DropMong 상세 페이지 조회에 적용할지는 프론트엔드 구현 비용과 어뷰징 위험도를 함께 고려해 결정해야 한다. 지금은 결정하지 않고 열어둔다.
