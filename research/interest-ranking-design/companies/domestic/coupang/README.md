# 쿠팡 (Coupang)

## 개요

- 구분: 국내 대형 이커머스, 마켓플레이스+직접판매 혼합 구조
- 주요 서비스: 상품 상세 페이지 데이터 서빙 (Catalog/Pricing/Fulfillment 등 여러 마이크로서비스 조합)
- 조사 초점: 여러 마이크로서비스에 흩어진 데이터를 read-heavy 트래픽에서 어떻게 빠르게 합쳐 서빙하는가

## 참고할 점

- interest-service도 "찜/조회수 랭킹"이라는 비교적 정확도 허용 범위가 넓은 데이터와, "재고 소진율" 같은 즉시성이 중요한 데이터를 같은 캐시 정책으로 묶지 않는 것이 좋다는 근거가 된다. 오픈 전 랭킹(찜+조회수)은 분 단위 캐시로도 충분할 수 있지만, 오픈 후 랭킹(소진율)은 더 짧은 갱신 주기가 필요할 수 있다.
- 여러 마이크로서비스 데이터를 큐로 모아 공용 스토리지에서 합쳐 서빙하는 패턴은, interest-service가 catalog-service/order-service의 이벤트를 구독해 자체 read model을 유지하는 현재 설계 방향과 같은 맥락이다.

## DropMong에 그대로 가져오면 안 되는 점

- 쿠팡의 "코어 서빙 레이어"는 조직 규모가 다른 여러 팀의 데이터를 통합하는 대형 인프라 투자다. DropMong/interest-service는 초기 스코프에서 이 정도 규모의 통합 계층을 새로 만들 필요는 없고, 필요한 이벤트만 구독해 자체 저장소에 반영하는 가벼운 방식으로 충분하다.

## 확인 필요

- CSP/N-CSP 클러스터 분리, read-through 캐시의 정확한 구현 기술(어떤 캐시 제품인지)은 원문 요약 수준에서는 확인하지 못했다.
