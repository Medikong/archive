# 29CM

## 개요

- 구분: 국내 라이프스타일/패션 커머스
- 주요 서비스: 29CM
- 조사 초점: "좋아요"(찜류 기능)를 모놀리식에서 마이크로서비스로 분리한 무중단 마이그레이션 사례

## 핵심 설계 포인트

| 주제 | 확인한 내용 | 출처 |
| --- | --- | --- |
| 기능 분리 배경 | 비즈니스 확장에 대응하기 위해 Monolithic에서 쓰던 좋아요 API를 V2로 전환 | [API V2 전환과 DB 무중단 마이그레이션 후기](https://medium.com/29cm/api-v2-%EC%A0%84%ED%99%98%EA%B3%BC-db-%EB%AC%B4%EC%A4%91%EB%8B%A8-%EB%A7%88%EC%9D%B4%EA%B7%B8%EB%A0%88%EC%9D%B4%EC%85%98-%ED%9B%84%EA%B8%B0-8b39eb0db566) |

## 참고할 점

- 좋아요(찜류) 기능이 처음엔 모놀리식에 있다가 트래픽/조직 확장으로 결국 독립 서비스로 옮겨진 사례다. DropMong은 이 과정을 미리 겪지 않기 위해 **찜 기능을 처음부터 interest-service로 독립**시키는 근거로 삼는다.
- 무중단 마이그레이션에서 핵심은 "마이그레이션 도중 변경되는 데이터를 어떻게 동기화하는가"였다. 이는 DropMong이 나중에 interest-service의 저장소나 구조를 바꿀 때도 참고할 수 있는 일반 원칙이다.

## DropMong에 그대로 가져오면 안 되는 점

- 29CM 사례는 이미 존재하는 대규모 서비스의 마이그레이션 상황이다. DropMong interest-service는 그린필드로 시작하므로 SQS 이중 기록 같은 마이그레이션 전용 기법은 지금 단계에서 필요 없다. 이 사례는 "왜 처음부터 분리하는가"의 근거로만 쓴다.

## 확인 필요

- 마이그레이션 시점의 실제 트래픽 규모, 장애 발생 여부는 스니펫만으로 확인하지 못했다.
