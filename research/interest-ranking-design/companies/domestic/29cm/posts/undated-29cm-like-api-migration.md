- 주제: 좋아요 API, 마이크로서비스 분리, 무중단 마이그레이션, SQS 이중 기록
- 확인 방식: WebSearch 결과 스니펫 기준 (원문 전체 미열람)

## 원문 확인 내용

검색 스니펫 기준으로 다음을 확인했다.

- 29CM은 비즈니스 확장에 대응하기 위해 Monolithic에서 쓰던 좋아요 API를 V2로 전환했다.
- 데이터는 PostgreSQL(V1)에서 Microservice가 사용하는 MySQL(V2)로 무중단 마이그레이션했다.
- 마이그레이션 도중 변경되는 데이터를 동기화하기 위해, V1 API에서 V1 테이블에 저장한 값을 JSON으로 encoding해 SQS Queue에 적재했다.
- 기존 V1 테이블에서 V2 테이블로 데이터 마이그레이션을 진행한 뒤, 완료 시점 이후 SQS Queue에 쌓여있던 변동사항을 순차적으로 V2 테이블에 반영했다.

## 적용 아이디어

- 찜(좋아요류) 기능을 다른 서비스에 임시로 얹어두면, 나중에 독립 서비스로 옮길 때 이런 이중 기록/동기화 비용이 발생한다. DropMong은 이 비용을 피하기 위해 interest-service를 처음부터 독립 서비스로 시작한다 (`REQ.A.07.PP-001`).

## 확인 필요

- 마이그레이션 규모(사용자 수, 데이터 건수), 소요 기간, 장애 발생 여부는 이 스니펫만으로 확인할 수 없다.
