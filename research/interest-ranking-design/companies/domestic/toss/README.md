# Toss

## 개요

- 구분: 국내 핀테크, 대규모 실시간 트래픽(라이브 쇼핑, 포인트 지급) 처리 사례
- 주요 서비스: 토스 라이브 쇼핑, 포인트 지급 API
- 조사 초점: 찜/조회수 같은 "카운터성 데이터"를 서버 증설 없이 처리하는 캐싱/집계 전략

## 핵심 설계 포인트

| 주제 | 확인한 내용 | 출처 |
| --- | --- | --- |
| 캐시 데이터 분류 | 모든 유저에게 동일한 "Universal Data"(방송 리스트 등)와 유저별로 다른 "User-Specific Data"(시청 여부 등)를 구분해 다르게 캐싱 | [서버 증설 없이 처리하는 대규모 트래픽](https://toss.tech/article/27600) |
| 핫키 완화 | Universal Data는 Redis 대신 웹 서버 로컬 캐시에 전부 올리고, 변경 시 Redis Pub/Sub으로 각 서버 로컬 캐시를 동기화 | 위 출처 |
| 카운터 처리 | 선착순/카운팅 로직에서 Redis INCR을 매 요청마다 부르지 않고, 로컬 캐시에서 먼저 카운팅한 뒤 일정 주기로 ScheduleJob이 Redis에 flush | 위 출처 |
| API 통합 | 조회 관련 API 3개를 1개(`/view`)로 통합해 피크 트래픽을 50% 감소시킴 | 위 출처 |

## 참고할 점

- DropMong의 "찜 수/조회수" 랭킹 점수도 오픈 순간 특정 인기 드롭 하나에 요청이 몰리는 **핫키 문제**를 겪을 수 있다. Toss의 "로컬 캐시 우선 집계 + 주기적 Redis flush" 패턴은 매 조회/찜마다 Redis에 직접 쓰는 대신 완충 지대를 두는 방법으로 참고할 수 있다.
- "Universal Data는 로컬 캐시, User-Specific Data는 압축된 Redis"라는 구분은, interest-service가 랭킹 결과(모두에게 동일)와 개인별 찜 여부(사용자마다 다름)를 다른 전략으로 캐싱해야 한다는 설계 힌트로 이어진다.

## DropMong에 그대로 가져오면 안 되는 점

- 로컬 캐시 우선 집계는 정확도(약간의 지연/유실 허용)와 처리량을 맞바꾸는 방식이다. DropMong의 랭킹 점수는 실시간성이 완전히 정확할 필요는 없지만, 찜 자체의 성공/실패 응답은 사용자에게 즉시 정확하게 보여야 하므로 두 데이터(찜 상태 자체 vs 랭킹 집계용 카운트)를 다른 정합성 수준으로 다뤄야 한다.

## 확인 필요

- ScheduleJob의 flush 주기, 로컬 캐시 유실 시 데이터 손실 허용 범위 등 구체적인 수치는 원문에서 확인하지 못했다.
