# Taxonomy

찜/실시간 인기 랭킹/조회수 설계 사례를 읽을 때 사용할 분류 기준이다.

## 핵심 관점

| 분류 | 확인할 질문 |
| --- | --- |
| 관심 신호 종류 | 어떤 사용자 행동(조회, 찜, 거래, 공유, 체류시간)을 인기/관심 신호로 쓰는가? |
| 신호 집계 방식 | 원시 카운트, 가중합, 최근 평균 대비 증가율(baseline 비교) 중 무엇을 쓰는가? |
| 시간 감쇠 | 오래된 활동을 어떻게 깎아내리는가? (선형, 로그 스케일, 거듭제곱 감쇠) |
| 중복/어뷰징 방지 | 동일 사용자·기기의 반복 행동을 어떻게 걸러내는가? |
| 랭킹 기간 구분 | 실시간/일간/월간처럼 여러 기간의 랭킹을 어떻게 나누는가? |
| 신규/기존 구분 | 새로 나온 항목과 기존 항목을 같은 기준으로 비교하는가, 분리하는가? |
| 공정성/설계 철학 | 랭킹이 조작되지 않는다는 신뢰를 사용자에게 어떻게 보여주는가? |
| 조회수 정의 | "본 것"을 어떤 기준(체류시간, 행동, 재생시간)으로 정의하는가? |
| 기능 분리 시점 | 관심/찜 기능을 언제 별도 서비스로 분리하는가, 분리 비용은 무엇인가? |

## 기록 방식

각 포스트나 회사별 정리는 다음 질문에 답할 수 있게 작성한다.

1. 이 글이 다루는 인기/관심 신호 문제는 무엇인가?
2. 선택한 신호 구성이나 공식은 무엇이며, 왜 그 방식이 필요한가?
3. 정확성(신뢰성)과 성능/비용 사이의 절충점은 무엇인가?
4. 어뷰징이나 조작을 어떻게 방지하는가?
5. DropMong 설계에 직접 참고할 점과 그대로 가져오면 안 되는 점은 무엇인가?

## 증거 수준

| 수준 | 의미 |
| --- | --- |
| 원문 확인 | 공개 글/제품 화면에서 직접 확인한 내용 |
| 문맥 추정 | 원문 조합으로 추정 가능하지만 직접 설명되지는 않은 내용 |
| 적용 아이디어 | DropMong에 참고할 수 있는 해석 또는 제안 |

분석 문서에서는 세 수준을 섞지 않는다. 추정이나 적용 아이디어는 명시적으로 구분한다.

이번 조사는 WebSearch 결과 스니펫 기준으로 진행했다. 원문 전체를 열람하지 못한 항목은 "원문 확인"이 아니라 "스니펫 확인"으로 별도 표기한다.
