# Taxonomy

인증 서비스 설계 사례를 읽을 때 사용할 분류 기준이다.

## 핵심 관점

| 분류 | 확인할 질문 |
| --- | --- |
| 로그인 | 사용자는 어떤 식별자와 증명 수단으로 로그인하는가? |
| 세션 | 로그인 상태는 서버 세션, 클라이언트 토큰, 하이브리드 중 무엇으로 유지하는가? |
| 토큰 | access token, refresh token, rotation, revocation을 어떻게 다루는가? |
| OAuth/OIDC | 외부 IdP와 내부 사용자 계정을 어떻게 연결하는가? |
| MFA | 추가 인증은 어떤 조건에서 요구하는가? |
| 위험 기반 인증 | 기기, 위치, 행위, 거래 위험도를 인증 강도에 반영하는가? |
| 계정 연결 | 동일 사용자의 여러 로그인 수단을 어떻게 병합하거나 분리하는가? |
| 내부 서비스 인증 | 서비스 간 호출은 JWT, mTLS, API key, workload identity 중 무엇을 쓰는가? |
| 인가 | 인증 결과가 권한, 역할, 정책 평가로 어떻게 이어지는가? |
| 감사 로그 | 로그인, 토큰 발급, 권한 변경, 실패 이벤트를 어떻게 기록하는가? |
| 운영/보안 | 키 회전, 장애 대응, rate limit, abuse detection을 어떻게 운영하는가? |

## 기록 방식

각 포스트나 회사별 정리는 다음 질문에 답할 수 있게 작성한다.

1. 이 글이 다루는 인증 문제는 무엇인가?
2. 선택한 구조는 무엇이며, 왜 그 구조가 필요한가?
3. 사용자 경험과 보안 사이의 절충점은 무엇인가?
4. 장애, 공격, 운영 부담을 어떻게 줄이는가?
5. DropMong 설계에 직접 참고할 점과 그대로 가져오면 안 되는 점은 무엇인가?

## 증거 수준

| 수준 | 의미 |
| --- | --- |
| 원문 확인 | 공개 글에서 직접 확인한 내용 |
| 문맥 추정 | 원문 조합으로 추정 가능하지만 직접 설명되지는 않은 내용 |
| 적용 아이디어 | DropMong에 참고할 수 있는 해석 또는 제안 |

분석 문서에서는 세 수준을 섞지 않는다. 추정이나 적용 아이디어는 명시적으로 구분한다.
