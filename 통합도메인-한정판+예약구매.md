# 통합 도메인 분석: 한정판 + 예약구매 커머스
> 평가표 (1) 도메인·아키텍처 정립 — 세부과제 1: MSA 경계 정의
> 작성일: 2026-07-02

---

## 1. 도메인 정의

**"한정 수량 상품을 즉시판매(한정판) 또는 사전예약판매(예약구매) 두 방식으로 운영하는 커머스 플랫폼"**

실제 서비스에서도 흔한 조합임 — 스니커즈 리셀 플랫폼이 한정판 드롭도 하고 사전예약(래플) 판매도 함께 운영하는 것처럼, 하나의 플랫폼이 상품 특성에 따라 **판매 방식(saleType)** 을 다르게 가져가는 구조.

| 판매 방식 | 흐름 | 예시 |
|---|---|---|
| **즉시판매 (IMMEDIATE)** | 재고 있음 → 결제 → 즉시 출고 | 한정판 스니커즈 드롭 |
| **예약판매 (PREORDER)** | 재고 없음(생산전) → 예약금 결제 → 생산완료 → 잔금 결제 → 출고 | 게임 콜렉터 에디션 사전예약 |

같은 플랫폼, 같은 서비스 집합 안에서 **주문 타입에 따라 분기**하는 게 핵심 설계 포인트.

---

## 2. 선정 근거

### 왜 합치는가
A(한정판)와 B(예약구매)를 따로 만들면 catalog·payment·order 로직이 대부분 겹치는데도 서비스를 중복 구축하게 됨. 반면 하나로 합치면:
- **재사용성**: catalog·order·payment의 80% 로직을 공유하고, 판매방식별 차이만 분기 처리
- **실전성**: 실제 커머스 플랫폼은 이렇게 여러 판매방식을 한 시스템에서 지원하는 경우가 많아 "실무에 가까운 설계"라는 인상을 줄 수 있음
- **평가 관점**: 하나의 서비스 집합 안에서 두 가지 트랜잭션 흐름을 도메인 모델로 어떻게 분기·수용했는지 보여주는 것 자체가 설계 역량 증명 포인트가 됨

### 티켓몽 자산 매핑
| 티켓몽 개념 | 통합 도메인 개념 |
|---|---|
| 콘서트 (concert) | 상품 (product, saleType 보유) |
| 좌석 즉시 예약 | 즉시판매 HOLD (초 단위 TTL) |
| 콘서트 사전 예매 | 예약판매 HOLD (긴 TTL, 생산완료까지) |
| 좌석 상태 HELD→ISSUING→TICKETED | 주문 상태 HELD→(즉시:PAID→CONFIRMED / 예약:DEPOSIT_PAID→BALANCE_PAID→CONFIRMED) |

reservation-service가 이미 "짧은 TTL 즉시 확보"와 "긴 TTL 사전 확보" 두 성격을 다 다뤄본 경험이 있어, 통합 설계에도 그대로 응용 가능.

## 3. 서비스 분리안 (5개)

### catalog — 통합 상품 카탈로그 서비스
**책임**: 상품 정보 + **saleType(IMMEDIATE/PREORDER)** 필드로 판매방식 구분, 드롭/예약 오픈 일정, 가격(즉시가 또는 예약금+잔금 구성)

**주요 API**: `GET /products`, `GET /products/{id}` (응답에 saleType 포함)

**데이터 모델 핵심**:
```
Product {
  id, name, category, price,
  saleType: IMMEDIATE | PREORDER,
  saleOpenAt, saleCloseAt,
  preorderConfig: { depositAmount, expectedShipDate } // PREORDER일 때만 사용
}
```

**재활용도**: 신규 구축, concert-service 참고

---

### inventory — 통합 재고/슬롯 서비스 핵심 차별화
**책임**: 판매방식에 따라 **HOLD 정책을 다르게 적용**하는 것이 이 서비스의 핵심 설계 지점

| 구분 | 즉시판매 HOLD | 예약판매 HOLD |
|---|---|---|
| TTL(임시확보 유지시간) | 짧음 (예: 5분, 결제창 시간) | 김 (예: 생산완료 시점까지, 며칠~몇달) |
| 재고 원본 | 실제 창고 재고 수량 | 예약 가능 수량(생산 예정 캡) |
| 만료 시 처리 | 즉시 RELEASE, 재고 복구 | 예약금 미결제 시 RELEASE |

**주요 API**: `POST /inventory/{sku}/hold?type=IMMEDIATE|PREORDER`, `POST /inventory/{sku}/confirm`, `POST /inventory/{sku}/release`

**재활용도**: reservation-service의 좌석 HELD 로직(짧은 TTL)과 콘서트 사전예매 대기열 로직(긴 TTL) 둘 다 이미 존재 → 하나의 서비스에서 TTL 파라미터만 다르게 주는 방식으로 통합 재활용 가능

---

### order — 통합 주문 서비스 핵심 차별화
**책임**: saleType에 따라 분기되는 단일 상태기계 운영

```
공통 시작: CREATED → HELD

IMMEDIATE 분기:
  HELD → PAID → CONFIRMED → SHIPPED

PREORDER 분기:
  HELD → DEPOSIT_PAID → (생산대기) → BALANCE_DUE
       → BALANCE_PAID → CONFIRMED → SHIPPED
```

**주요 API**: `POST /orders`, `GET /orders/{id}`, `PATCH /orders/{id}/cancel`

**설계 포인트**: 상태기계를 하나로 유지하되 saleType 필드로 분기 처리 — 두 개의 별도 서비스로 쪼개면 상태 동기화가 오히려 더 복잡해지므로 **단일 서비스 + 내부 분기**가 더 나은 설계

**재활용도**: reservation-service의 상태 전이 로직 구조를 확장해서 분기 추가

---

### payment — 통합 결제 서비스
**책임**: 단일결제(즉시판매)와 분할결제(예약구매) 둘 다 지원

**주요 API**:
- `POST /payments` (즉시판매: 전액 결제)
- `POST /payments/deposit` (예약구매: 예약금)
- `POST /payments/balance` (예약구매: 잔금)
- `POST /payments/{id}/refund`

**재활용도**: payment-service 그대로 재활용 + 분할결제 엔드포인트만 추가

---

### fulfillment — 출고 서비스
**책임**: 즉시판매는 결제 즉시 출고 트리거, 예약구매는 생산완료+잔금결제 완료 후 출고 트리거

**주요 API**: `POST /fulfillments`, `PATCH /fulfillments/{id}/ship`

**재활용도**: 낮음(신규), 다만 트리거 조건만 분기 처리라 로직 자체는 단순

---

### notification (선택, 6번째)
**책임**: 판매방식별로 다른 알림 시나리오 — 즉시판매는 "결제 완료", 예약구매는 "예약 확정 → 잔금 결제 안내 → 출고 알림" 다단계

**재활용도**: 신규 구축, Kafka 인프라 재활용

---

## 4. 분리 이유 (Bounded Context 근거)

**catalog / inventory 분리**
- saleType과 무관하게 항상 유효한 원칙: 상품 정보(정적)와 재고·슬롯 상태(동적)는 변경 주기가 다름
- inventory는 두 가지 TTL 정책을 다루는 것 자체가 복잡한 관심사라 catalog와 분리해야 관리 가능

**order를 하나로 유지(분리하지 않음)**
- IMMEDIATE 전용 order와 PREORDER 전용 order로 나누는 대신 **하나의 order 서비스 안에서 saleType 분기**를 택함
- 이유: 두 흐름 모두 "주문 하나의 생애주기"라는 같은 개념적 루트(Aggregate Root)를 공유함 — 억지로 서비스를 쪼개면 오히려 두 서비스 간 상태 동기화 문제가 새로 생김 (Bounded Context는 무조건 잘게 쪼개는 게 아니라, "같은 것을 소유하는 단위"로 묶는 것이 원칙)

**payment 분리**
- 외부 PG 의존성 + 두 판매방식 모두에서 공통으로 필요한 기능이라 별도 서비스로 유지 (기존 A/B안과 동일한 논리)

**fulfillment 분리**
- 출고는 생산·물류라는 완전히 다른 외부 변수에 의존 — order의 결제 동시성 이슈와 무관한 관심사

## 5. 통신 방식

| 흐름 | 방식 | 이유 |
|---|---|---|
| order → inventory (HOLD 요청, type 파라미터 포함) | 동기 REST | 즉시판매는 오버셀 방지, 예약판매도 수량 캡 확인 필요 |
| order → payment (즉시결제 또는 예약금결제) | 동기 REST | 결제 성공 여부로 주문 상태 즉시 분기 |
| order → payment (잔금 청구, 예약판매만) | 비동기(Kafka) 트리거 + 동기 처리 | 생산완료 이벤트 수신 후 잔금 청구 프로세스 시작 |
| fulfillment → order (출고 완료 반영) | 비동기(Kafka) | 출고는 결제 흐름과 시간차가 크므로 이벤트로 충분 |
| order → notification | 비동기(Kafka) | 지연 허용 |

---

## 6. 이 설계의 장단점

**장점**
- 서비스 개수를 5~6개로 유지하면서 두 가지 판매 흐름을 모두 커버 — 평가표의 "3~5개 서비스" 기준에 정확히 부합
- catalog·payment는 완전 공유, inventory·order만 내부 분기 → 코드 중복 최소화
- 티켓몽의 "즉시 좌석확보"와 "사전 예매" 두 경험을 모두 재활용할 수 있어 재활용률이 가장 높음

**단점(설계 시 주의점)**
- order 서비스의 상태기계가 분기 때문에 다소 복잡해짐 — 상태 다이어그램을 명확히 문서화해두지 않으면 팀원 간 커뮤니케이션 비용 증가
- inventory의 HOLD 정책이 두 가지라 테스트 케이스가 늘어남 (부하테스트 시나리오도 판매방식별로 따로 설계 필요)
